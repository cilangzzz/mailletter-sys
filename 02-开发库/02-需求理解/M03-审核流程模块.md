## 1. 模块概述

### 1.1 模块定位

审核流程模块是gaxx-pro信件处理系统的核心业务流程控制模块，负责信件办理结果的审核与流转控制。该模块确保信件回复内容的质量，规范信件处理流程，保障群众诉求得到有效解决。

### 1.2 技术架构

本模块采用**模板方法模式（Template Method Pattern）**进行设计，通过统一的处理骨架确保流程的一致性和可维护性。

#### 1.2.1 处理器架构

系统定义了统一的处理流程骨架，每个处理器都遵循相同的处理步骤：

1. **准备上下文**：加载当前用户、部门、操作时间等信息
2. **加载信件数据**：根据信件ID获取信件详情
3. **执行前置校验**：验证权限、状态等前置条件
4. **执行具体业务逻辑**：各处理器实现各自的业务逻辑
5. **创建流程记录**：记录操作日志
6. **更新关联数据**：更新相关实体状态
7. **执行后置任务**：异步执行通知等任务
8. **构建返回结果**：返回处理结果

#### 1.2.2 处理器列表

| 处理器名称 | 处理器类型 | 功能说明 |
|------------|------------|----------|
| ReplyHandler | REPLY | 提交回复处理器 |
| HandleHandler | HANDLE | 提交办理处理器 |
| TransferHandler | TRANSFER | 指派办理单位处理器 |
| StepTransferHandler | STEP_TRANSFER | 逐级指派处理器 |
| CancelTransferHandler | CANCEL_TRANSFER | 取消指派处理器 |
| StepCancelTransferHandler | STEP_CANCEL_TRANSFER | 逐级取消指派处理器 |
| RollbackCantHandleHandler | ROLLBACK_CANT_HANDLE | 无法处理退回处理器 |
| RollbackHandler | ROLLBACK | 退回重新修改/重新办理处理器 |
| ApprovalPassHandler | APPROVAL_PASS | 审核通过处理器 |
| ApprovalNoHandler | APPROVAL_NO | 审核不通过处理器 |
| ReassignHandler | REASSIGN | 改派处室处理器 |

### 1.3 业务价值

- **质量保障**：通过多级审核机制，确保回复内容的准确性、完整性和规范性
- **流程规范**：明确审核流程节点和操作权限，规范处理行为
- **效率提升**：提供灵活的审核路径（一审+终审、直接终审），适应不同业务场景
- **责任追溯**：完整的流程记录，支持操作追溯和责任认定
- **可扩展性**：模板方法模式支持新增流程处理器的快速扩展

## 2. 功能点详细分析

### 2.1 F03-01 提交回复

#### 功能描述

主办单位完成信件调查处理后，提交回复内容进入审核流程。回复内容需经过审核通过后方可发送给网民。

#### 操作角色

| 角色 | 操作权限 | 说明 |
|------|----------|------|
| 主办单位 | 可提交回复 | 负责信件办理并提交回复内容 |

#### 输入

| 参数名称 | 参数类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 信件编号 | 编号 | 是 | 信件唯一标识 |
| 回复内容 | 正文 | 是 | 给网民的回复正文 |
| 上级回复内容 | 正文 | 否 | 上级部门的回复内容（如有） |
| 是否结案 | 标记 | 否 | 0否/1是，是否建议结案 |
| 回复附件 | 文件 | 否 | 回复相关附件材料 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 回复记录 | 创建的回复记录信息 |
| 流程状态 | 信件状态变更为"待审核" |
| 流程日志 | 记录提交操作日志 |
| 办理单位状态更新 | 更新办理单位的回复状态 |

#### 业务规则

1. **前置状态检查**：信件状态必须为"受理中"
2. **权限验证**：必须是信件的主办单位才能提交回复
3. **内容校验**：回复内容不能为空，需符合格式规范
4. **唯一性校验**：同一办理单位同一信件只能有一个有效回复记录
5. **状态变更**：提交成功后信件状态变更为"待审核"
6. **时限检查**：提交时检查是否超期，超期需说明原因
7. **举报公安民警件特殊处理**：统一变成提交办理表
8. **多次回复校验**：如果已答复，要进行多次答复，则需校验当前信件是否已结案

#### 前置条件

- 信件已指派主办单位
- 当前用户为主办单位人员
- 信件状态为"受理中"
- 已完成调查核实工作

#### 异常处理

| 异常场景 | 错误码 | 处理方式 |
|----------|--------|----------|
| 信件不存在 | LETTER_NOT_EXISTS | 返回错误提示"信件不存在" |
| 当前单位已被取消指派 | CURRENT_UNIT_HAD_BEEN_CANCEL_ASSIGN | 返回错误提示"当前单位已被取消指派" |
| 当前单位未提交办理表 | CURRENT_UNIT_HAD_NOT_SUBMIT_HANDLE | 返回错误提示"当前单位未提交办理表" |
| 信件已完成 | LETTER_HAD_BEEN_COMPLETE | 返回错误提示"信件已完成" |

#### 输入

|----------|----------|----------|------|

| **回访信息** | | | |
| 是否回访 | 标记 | 是 | 是否已回访群众 |
| 未回访原因 | 原因 | 否 | 未回访的具体原因 |
| 回访情况 | 说明 | 否 | 回访情况详细说明 |
| 群众电话回访反映 | 说明 | 否 | 群众电话回访反映内容 |
| 是否需要再次回复 | 标记 | 否 | 是否需要再次回复 |
| **满意度评价** | | | |
| 诉求解决满意度 | 评分 | 否 | 满意/基本满意/不满意 |
| 响应速度满意度 | 评分 | 否 | 满意/基本满意/不满意 |
| 服务态度满意度 | 评分 | 否 | 满意/基本满意/不满意 |
| 办事能力满意度 | 评分 | 否 | 满意/基本满意/不满意 |
| 跟进服务满意度 | 评分 | 否 | 满意/基本满意/不满意 |
| 整体满意度 | 评分 | 否 | 满意/基本满意/不满意 |
| **调查情况** | | | |
| 说法是否一致 | 标记 | 否 | 公安机关和群众说法是否一致 |
| 调查情况 | 状态 | 否 | 调查完成情况 |
| 无法调查原因 | 原因 | 否 | 无法开展调查的原因 |
| 解决情况 | 状态 | 否 | 问题解决情况说明 |

| 接访时间 | 时间 | 否 | 接访时间 |

| 办理人姓名 | 人员 | 是 | 主办民警姓名 |

| 办理人单位 | 单位 | 是 | 主办民警所在单位 |

| 输出项 | 说明 |
|--------|------|
| 办理记录 | 创建的办理记录信息 |
| 流程记录 | 创建流程记录 |
| 办理单位状态更新 | 更新办理单位的办理状态 |

#### 业务规则

1. **关联校验**：办理单位编号必须有效且属于当前用户所在单位
2. **必填校验**：办理人信息为必填项
3. **逻辑校验**：如选择"未回访"，必须填写未回访原因
4. **满意度完整性**：如已回访，建议填写满意度评价
5. **接访校验**：如选择"已接访"，必须填写接访方式和接访领导
6. **审核信息**：重大案件需呈法制或督察部门审核
7. **重复提交校验**：避免重复提交办理，举报公安民警部门可以重复提交办理
8. **办理时间更新**：多次办理要更新办理时间，而不是创建时间

#### 前置条件

- 信件已指派主办单位
- 当前用户为主办单位人员
- 办理单位状态为"已指派"

#### 异常处理

| 异常场景 | 错误码 | 处理方式 |
|----------|--------|----------|
| 办理单位不存在 | LETTER_NOT_EXISTS | 返回错误提示"办理单位记录不存在" |
| 权限不足 | CURRENT_UNIT_HAD_BEEN_CANCEL_ASSIGN | 返回错误提示"当前单位已被取消指派" |
| 重复提交办理 | REPLY_PROCESSING_REPEAT_HANDLE | 返回错误提示"请勿重复提交办理" |

### 2.3 F03-03 指派办理单位

#### 功能描述

接收单位为信件指派办理单位，指定其角色（主办/督办/协办），明确信件的办理责任主体。

#### 操作角色

| 角色 | 操作权限 | 说明 |
|------|----------|------|
| 接收单位 | 可指派办理单位 | 负责信件分配 |
| 中间单位 | 可逐级指派 | 可向下级继续指派 |

#### 输入

| 参数名称 | 参数类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 信件编号 | 编号 | 是 | 信件唯一标识 |
| 主办单位列表 | 单位列表 | 是 | 被指派为主办的部门列表 |
| 协办单位列表 | 单位列表 | 否 | 被指派为协办的部门列表 |
| 督办单位列表 | 单位列表 | 否 | 被指派为督办的部门列表 |
| 流程状态 | 状态 | 是 | 指派后的处理状态 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 办理单位记录 | 创建的办理单位记录 |
| 流程记录 | 创建指派流程记录 |
| 信件数据权限更新 | 更新信件的部门路径 |
| 受理编号生成 | 首次指派时生成受理编号 |
| 受理时间生成 | 首次指派时生成受理时间 |

#### 业务规则

1. **主办必填**：每封信件必须有主办单位
2. **角色限制**：
   - 主办单位：负责信件的主答复工作，可有多个
   - 督办单位：负责监督办理进度，可有多个
   - 协办单位：协助办理相关工作，可有多个
3. **数据权限**：指派后更新信件的部门路径，授予被指派单位数据权限
4. **状态联动**：
   - 转给本站点且没有下级部门时，信件状态变为"受理中"
   - 有下级部门时，状态保持不变
5. **受理编号**：首次指派时生成受理编号
6. **受理时间**：首次指派时生成受理时间
7. **接收单位跳过**：不创建接收单位的办理单位记录

#### 前置条件

- 信件已创建且处于"待处理"或"受理中"状态
- 操作用户有指派权限
- 待指派的办理单位在系统中有效且可用

#### 异常处理

| 异常场景 | 错误码 | 处理方式 |
|----------|--------|----------|
| 信件不存在 | LETTER_NOT_EXISTS | 返回错误提示"信件不存在" |
| 当前单位已被取消指派 | CURRENT_UNIT_HAD_BEEN_CANCEL_ASSIGN | 返回错误提示"当前单位已被取消指派" |

### 2.4 F03-04 逐级指派

#### 功能描述

支持多级单位逐级向下指派信件，实现信件的分级流转。

#### 操作角色

| 角色 | 操作权限 | 说明 |
|------|----------|------|
| 接收单位 | 可逐级指派 | 可向下级单位指派 |
| 中间单位 | 可逐级指派 | 可继续向下级指派 |

#### 输入

| 参数名称 | 参数类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 信件编号 | 编号 | 是 | 信件唯一标识 |
| 逐级指派列表 | 列表 | 是 | 包含多个指派信息的列表 |
| - 来源单位 | 单位 | 是 | 发起指派的单位 |
| - 被指派单位 | 单位 | 是 | 被指派的下级单位 |
| - 部门类型 | 类型 | 是 | 主办/督办/协办 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 办理单位记录 | 创建多个办理单位记录 |
| 流程记录 | 创建多个指派流程记录 |
| 中间单位自动受理 | 涉及下级单位被指派时，上级中间单位自动受理 |

#### 业务规则

1. **批量指派**：支持一次操作指派多个单位
2. **中间单位自动受理**：涉及指派的流程，只要有下级单位被指派，作为上级单位且是中间单位自动受理
3. **待回复状态**：被指派单位的初始状态为"待回复"
4. **数据权限更新**：更新信件的部门路径

#### 前置条件

- 信件已创建
- 操作用户有指派权限
- 被指派单位是当前单位的下级单位

#### 异常处理

| 异常场景 | 错误码 | 处理方式 |
|----------|--------|----------|
| 信件不存在 | LETTER_NOT_EXISTS | 返回错误提示"信件不存在" |
| 权限不足 | CURRENT_UNIT_HAD_BEEN_CANCEL_ASSIGN | 返回错误提示"当前单位已被取消指派" |

### 2.5 F03-05 取消指派

#### 功能描述

取消已指派的办理单位，使用BFS（广度优先搜索）算法递归取消所有子级单位的指派。

#### 操作角色

| 角色 | 操作权限 | 说明 |
|------|----------|------|
| 接收单位 | 可取消指派 | 可取消下级单位的指派 |
| 上级单位 | 可取消指派 | 可取消自己指派的单位 |

#### 输入

| 参数名称 | 参数类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 信件编号 | 编号 | 是 | 信件唯一标识 |
| 取消单位 | 单位 | 是 | 要取消指派的单位 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 流程记录 | 创建取消指派的流程记录 |
| 办理单位状态更新 | 更新办理单位状态为取消 |
| 办理表删除 | 删除旧办理表和新办理表记录 |
| 数据权限移除 | 移除被取消单位的数据权限 |
| 信件状态更新 | 如果所有办理单位都被取消，信件状态回退到"待受理" |

#### 业务规则

1. **不能取消自己**：不能取消自己的指派
2. **BFS递归取消**：使用广度优先搜索算法，递归取消所有子级单位的指派
3. **办理表删除**：同时删除旧办理表和新办理表记录
4. **数据权限移除**：移除被取消单位的数据权限
5. **状态回退**：如果所有办理单位都被取消，信件状态回退到"待受理"

#### 前置条件

- 信件已指派办理单位
- 操作用户有取消指派权限
- 不能取消自己的指派

#### 异常处理

| 异常场景 | 错误码 | 处理方式 |
|----------|--------|----------|
| 信件不存在 | LETTER_NOT_EXISTS | 返回错误提示"信件不存在" |
| 不能取消自己的指派 | REPLAY_PROCESSING_CANT_CANCEL_SELF | 返回错误提示"不能取消自己的指派" |
| 当前单位已被取消指派 | CURRENT_UNIT_HAD_BEEN_CANCEL_ASSIGN | 返回错误提示"当前单位已被取消指派" |

| 办理表删除 | 删除相关办理表记录 |
| 数据权限移除 | 移除被取消单位的数据权限 |
| 信件状态更新 | 如果所有办理单位都被取消，信件状态回退到"待受理" |

#### 业务规则

1. **批量取消**：支持一次操作取消多个单位
2. **办理表删除**：同时删除旧办理表和新办理表记录
3. **数据权限移除**：移除被取消单位的数据权限
4. **状态回退**：如果所有办理单位都被取消，信件状态回退到"待受理"

#### 前置条件

- 信件已指派办理单位
- 操作用户有取消指派权限

### 2.7 F03-07 下级单位退回-无法处理

#### 功能描述

办理单位因客观原因无法处理该信件时，可申请退回上级单位。退回需说明具体原因，上级单位决定后续处理方式。

#### 操作角色

| 角色 | 操作权限 | 说明 |
|------|----------|------|
| 主办单位 | 可申请退回 | 因无法处理申请退回 |

#### 输入

| 参数名称 | 参数类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 信件编号 | 编号 | 是 | 信件唯一标识 |
| 退回原因 | 原因 | 是 | 无法处理的具体原因 |
| 退回类型 | 类型 | 是 | CANT_HANDLE（无法处理） |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 流程状态 | 信件退回至上级单位 |
| 流程记录 | 记录退回操作日志 |
| 数据权限移除 | 去除该部门的数据权限 |
| 信件状态更新 | 如果全部退回，信件状态变为"待处理" |

#### 业务规则

1. **状态检查**：信件状态必须为"受理中"
2. **权限校验**：只有主办单位可申请退回
3. **原因必填**：退回原因必须详细填写
4. **上级接收**：退回后信件流转至指派来源单位
5. **全部退回检测**：检查是否所有指派的单位都已退回
6. **状态变更**：如果全部退回，信件状态变更为"待处理"

#### 前置条件

- 信件已指派办理单位
- 当前用户为主办单位人员
- 信件状态为"受理中"
- 尚未提交回复

#### 异常处理

| 异常场景 | 错误码 | 处理方式 |
|----------|--------|----------|
| 信件不存在 | LETTER_NOT_EXISTS | 返回错误提示"信件不存在" |
| 必须选择退回部门 | LETTER_ROLL_BACK_DEPT_MUST_CHOOSE | 返回错误提示"必须选择退回部门" |

### 2.8 F03-08 上级单位退回-重新修改

#### 功能描述

审核单位审核回复内容后，认为需要修改完善，将信件退回主办单位要求修改回复内容或重新办理。主办单位修改后可再次提交审核。

#### 操作角色

| 角色 | 操作权限 | 说明 |
|------|----------|------|
| 接收单位 | 可退回修改 | 终审时退回修改 |
| 督办单位 | 可退回修改 | 一审时退回修改 |

#### 输入

| 参数名称 | 参数类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 信件编号 | 编号 | 是 | 信件唯一标识 |
| 退回意见 | 意见 | 是 | 要求修改的具体意见 |
| 退回类型 | 类型 | 是 | ROLLBACK_REPLAY（退回答复）/ROLLBACK_HANDLE（退回办理） |
| 修改要点 | 说明 | 否 | 需要重点修改的内容 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 流程状态 | 信件状态变更为"重新修改"或"重新办理" |
| 流程记录 | 记录退回操作日志 |
| 通知消息 | 通知主办单位修改 |
| 回复状态更新 | 更新回复记录状态为"已退回" |
| 办理单位状态更新 | 更新办理单位的回复状态或办理状态 |

#### 业务规则

1. **状态检查**：信件状态必须为"待审核"
2. **权限校验**：接收单位或督办单位可执行
3. **意见必填**：退回意见必须详细说明需要修改的内容
4. **状态变更**：信件状态变更为"重新修改"或"重新办理"
5. **回复状态**：回复记录状态变更为"已退回"
6. **通知机制**：退回后自动通知主办单位
7. **重复退回校验**：校验是否重复退回

#### 前置条件

- 信件状态为"待审核"
- 当前用户为接收单位或督办单位人员
- 存在待审核的回复内容

#### 异常处理

| 异常场景 | 错误码 | 处理方式 |
|----------|--------|----------|
| 信件不存在 | LETTER_NOT_EXISTS | 返回错误提示"信件不存在" |
| 必须选择退回部门 | LETTER_ROLL_BACK_DEPT_MUST_CHOOSE | 返回错误提示"必须选择退回部门" |
| 重复退回 | REPLY_PROCESSING_REPEAT_ROLL_BACK | 返回错误提示"请勿重复退回" |

### 2.9 F03-09 审核通过

#### 功能描述

审核单位对主办单位提交的回复内容进行审核，审核通过后信件进入"待回复"状态，准备正式回复网民。支持一审通过和终审通过。

#### 操作角色

| 角色 | 操作权限 | 说明 |
|------|----------|------|
| 督办单位 | 可一审通过 | 负责一级审核 |
| 接收单位 | 可终审通过 | 负责最终审核 |

#### 输入

| 参数名称 | 参数类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 信件编号 | 编号 | 是 | 信件唯一标识 |
| 审核意见 | 意见 | 否 | 审核意见 |
| 审核状态 | 状态 | 是 | AUDIT_HEAD（一审）/AUDIT_FINAL（终审） |
| 是否结案 | 标记 | 否 | 0否/1是 |
| 是否建议结案 | 标记 | 否 | 0否/1是 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 流程状态 | 一审通过后进入终审阶段，终审通过后进入"待回复"状态 |
| 流程记录 | 记录审核通过操作 |
| 审核标记 | 回复记录标记审核通过 |
| 回复记录更新 | 创建或更新回复记录 |
| 信件状态更新 | 终审通过后更新信件的结案状态、回复时间、回复内容 |

#### 业务规则

1. **权限校验**：只有接收单位可执行审核操作
2. **审核标记**：回复记录标记"一审已通过"或"终审已通过"
3. **流程推进**：一审通过后进入终审阶段
4. **终审处理**：
   - 终审通过后信件状态变更为"待回复"
   - 设置回复时间、回复内容、回复单位
   - 更新信件的结案状态
5. **通知机制**：自动通知相关单位

#### 前置条件

- 信件状态为"待审核"
- 当前用户为督办单位或接收单位人员
- 存在待审核的回复内容

#### 异常处理

| 异常场景 | 错误码 | 处理方式 |
|----------|--------|----------|
| 无审核权限 | REPLY_PROCESSING_APPROVE_NOT_PERMIT | 返回错误提示"无审核权限" |

### 2.11 F03-11 改派处室

#### 功能描述

更换信件的接收单位，根据数据来源类型进行不同的处理。适用于信访件、省厅件、举报公安民警件等。

#### 操作角色

| 角色 | 操作权限 | 说明 |
|------|----------|------|
| 接收单位 | 可改派处室 | 负责单位调整 |

#### 输入

| 参数名称 | 参数类型 | 是否必填 | 说明 |
|----------|----------|----------|------|
| 信件编号 | 编号 | 是 | 信件唯一标识 |
| 数据来源 | 类型 | 是 | 数据来源类型（信访件/省厅件/举报公安民警件） |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 单位变更记录 | 记录单位变更信息 |
| 流程状态 | 根据改派情况更新信件状态 |
| 标签修改权移交 | 移交标签的修改权给新单位 |

#### 业务规则

4. **流程记录**：记录改派操作日志

#### 前置条件

- 信件状态不为"已完成"
- 当前用户有改派权限

#### 异常处理

```
    │       │       ├── 满意度评价
    │       │       └── 办理人信息
    │       │
    │       └── 办理状态
    │               ├── handleStatus: 办理状态
    │               ├── replyStatus: 回复状态
    │               └── processStatus: 整体流程状态
    │
    └── 流程记录（ReplyProcessingDO）（一对多）
            ├── processingType: 流程类型
            ├── processingStatus: 流程状态码
            ├── sourceDeptId: 来源部门
            ├── assignedDeptId: 被指派部门
            ├── deptType: 部门类型
            └── processingResult: 处理结果
```

## 6. 用户故事

### 6.1 主办单位用户故事

**US-M03-01：提交回复内容**

作为一名主办单位工作人员，我希望能够提交信件回复内容并进入审核流程，以便我的工作成果能够得到审核确认。

验收标准：
- 能够填写完整的回复内容
- 提交后信件状态变为"待审核"
- 系统记录提交时间和操作人
- 提交后收到确认通知

**US-M03-02：提交办理详情**

作为一名主办单位工作人员，我希望能够提交办理详情（包括走访、满意度等信息），以便审核人员了解办理过程。

验收标准：
- 能够填写完整的办理表信息
- 办理人信息为必填项
- 系统自动关联对应的信件和办理单位

**US-M03-03：处理退回修改**

作为一名主办单位工作人员，当我的回复被退回修改时，我希望能看到具体的修改意见，以便我能针对性地修改回复内容。

验收标准：
- 能看到退回意见和修改要点
- 修改后可再次提交审核
- 系统记录修改次数

**US-M03-04：申请无法处理**

作为一名主办单位工作人员，当我遇到无法处理的信件时，我希望能够申请退回上级，以便上级重新安排处理。

验收标准：
- 能够说明无法处理的原因
- 退回后收到确认通知
- 上级能收到退回提醒

### 6.2 督办单位用户故事

**US-M03-05：一级审核回复**

作为一名督办单位工作人员，我希望能够审核主办单位提交的回复内容，以便确保回复质量达标。

验收标准：
- 能查看回复内容和办理详情
- 能选择通过或不通过
- 不通过时能填写驳回意见
- 系统记录审核时间和结果

### 6.3 接收单位用户故事

**US-M03-06：最终审核回复**

作为一名接收单位工作人员，我希望能够进行终审审核，以便确认回复内容可以发送给网民。

验收标准：
- 能查看一审意见（如有）
- 能选择终审通过或不通过
- 通过后信件进入待回复状态
- 系统记录终审时间和结果

**US-M03-07：指派办理单位**

作为一名接收单位工作人员，我希望能够为信件指派办理单位，以便明确信件的办理责任。

验收标准：
- 能选择主办、督办、协办单位
- 能说明指派原因
- 被指派单位能收到通知
- 系统记录指派操作

**US-M03-08：取消指派**

作为一名接收单位工作人员，我希望能够取消已指派的办理单位，以便在必要时调整信件的处理安排。

验收标准：
- 能选择要取消的单位
- 取消时自动取消所有子级单位
- 被取消单位能收到通知
- 系统记录取消操作
