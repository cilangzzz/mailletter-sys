## 1. 产品概述

### 1.1 产品背景

公安系统日常接收大量群众信访件，来源渠道多样：
- 厅长信箱/局长信箱
- 信访件
- 12389举报平台
- 省级平台推送

传统处理方式存在以下问题：
- **效率低下**：纸质流转、人工分发，耗时费力
- **跟踪困难**：信件状态难以实时掌握，容易遗漏
- **协作不便**：多部门协作缺乏统一平台，信息不共享
- **监督缺失**：办理进度、质量难以有效监督
- **数据分散**：统计分析困难，难以支撑决策

### 1.2 产品定位

**定位**：统一的公安信访件数字化处理平台，实现信件全流程闭环管理。

### 1.3 目标用户

| 角色 | 职责 |
|------|------|
| 接收单位 | 接收信件、指派办理单位、审核办理结果、回复网民 |
| 主办单位 | 主要办理信件、提交办理结果 |
| 督办单位 | 监督办理进度、催办、审核主办单位结果 |
| 协办单位 | 协助办理、提供专业意见 |

### 1.4 产品目标

1. **流程标准化**：建立规范的信件接收→指派→办理→审核→回复流程
2. **时效可控化**：设置预警机制，确保信件按时办结
3. **质量规范化**：多级审核确保答复质量
4. **协作一体化**：主办、督办、协办角色明确，协作顺畅
5. **数据可视化**：提供多维度统计分析，支撑管理决策

## 3. 业务流程

### 3.1 信件处理主流程

```
网民写信 → 接收单位接收 → 指派办理单位处理 → 办理单位处理审核 → 审核后接收单位回复网民
```

### 3.2 信件状态流转

```
待处理 → 受理中 → 待审核 → 待回复 → 已完成
           ↓                    ↓
      无法受理            审核不通过 → 重新修改 → 重新办理
```

**状态分组**：
| 状态组 | 说明 |
|--------|------|
| 待受理 | 信件待处理 |
| 受理中 | 信件正在办理 |
| 待审核 | 信件等待审核 |
| 已完成 | 信件已办结 |

### 3.3 审核流程

| 操作 | 功能描述 |
|------|----------|
| 回复信件 | 提交回复内容，状态变为待审核 |
| 提交办理表 | 提交办理信息 |
| 下级单位退回 | 下级单位无法处理时退回 |
| 上级单位退回-重新修改 | 退回要求重新修改回复 |
| 上级单位退回-重新办理 | 退回要求重新办理 |
| 审核一审通过 | 一级审核通过 |
| 审核终审通过 | 最终审核通过 |
| 直接终审通过 | 跳过一审直接终审 |
| 审核不通过 | 审核驳回 |

## 5. 附录

### 5.1 业务规则摘要

| 规则名称 | 规则描述 |
|----------|----------|
| 重复信件判断 | 根据来信人身份证号+标题相似度判断重复 |
| 状态流转规则 | 状态只能按流程流转，不可跳跃 |
| 办理单位唯一性 | 同一信件同一部门只能被指派一次 |
| 主办单位必填 | 每封信件必须有且仅有一个主办单位 |
| 答复修改限制 | 已答复网民的回复不可修改 |
| 自动关联预警 | 信件创建时根据站点类型自动关联预警牌 |
| 时长计算规则 | 时长按工作日计算，排除节假日 |

### 5.2 高级检索功能概览

**已实现的筛选条件（35个）**：

| 类别 | 筛选条件 |
|------|----------|
| 来信人信息 | 留言ID、姓名、身份证、手机号、联系地址、归属地址、受理编号 |
| 信件内容 | 留言标题、回复内容、留言内容、留言备注 |
| 时间类 | 留言时间、回复时间 |
| 状态类 | 信件状态、结案状态、情况类型、是否重复件 |
| 办理单位 | 责任处室、责任类型、指派单位状态 |
| 类型分类 | 留言类型、业务类型、案件指向、留言来源 |
| 特殊标记 | 加急信件、再次回复、信件录入、自定义标签 |
| 其他 | 选登情况、警告牌状态、命中名单类型、满意度 |
